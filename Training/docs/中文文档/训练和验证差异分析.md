## 一、Rollout逻辑对比

### 1.1 训练Rollout流程

**代码位置**: `verl/trainer/ppo/ray_trainer.py:1068-1093`

```python
# 训练阶段
batch: DataProto = DataProto.from_single_dict(batch_dict)

# 添加uid
batch.non_tensor_batch["uid"] = np.array(
    [str(uuid.uuid4()) for _ in range(len(batch.batch))], dtype=object
)

gen_batch = self._get_gen_batch(batch)

# 设置meta_info
gen_batch.meta_info["global_steps"] = self.global_steps
# ⚠️ 注意: 训练时没有设置 validate=True

# 重复采样(n=8)
gen_batch = gen_batch.repeat(
    repeat_times=self.config.actor_rollout_ref.rollout.n, 
    interleave=True
)

# 生成序列
gen_batch_output = self.actor_rollout_wg.generate_sequences(gen_batch)
```

**关键特征**:
- ✅ 使用 `rollout.n=8` (从配置文件)
- ❌ **没有设置** `meta_info["validate"] = True`
- ❌ **没有设置** `meta_info["do_sample"]`
- ✅ 使用默认的采样参数 (temperature, top_p, top_k 从 rollout config)

test_batch = DataProto.from_single_dict(test_data)

- ✅ 使用验证专用的采样参数 (val_kwargs.temperature, top_p, top_k)

### 1.3 SGLang Rollout如何处理validate标志

**代码位置**: `verl/workers/rollout/sglang_rollout/sglang_rollout.py:688-718`

```python
def _batch_level_generate_sequences(self, prompts: DataProto, **kwargs):
    # 提取标志
    do_sample = prompts.meta_info.get("do_sample", True)
    is_validate = prompts.meta_info.get("validate", False)
    
    # 创建采样参数
    request_sampling_params = self.sampling_params.copy()
    
    if not do_sample:
        # 贪心解码 (do_sample=False)
        request_sampling_params.update({
            "n": 1,
            "temperature": 0,
            "top_p": 1,
            "top_k": -1,
            # ...
        })
    elif is_validate:
        # ⚠️ 验证模式: 使用val_kwargs
        request_sampling_params.update({
            "top_k": self.config.val_kwargs.top_k,
            "top_p": self.config.val_kwargs.top_p,
            "temperature": self.config.val_kwargs.temperature,
            "n": 1,  # 已经在ray_trainer中重复了
        })
    else:
        # 训练模式: 使用默认rollout config
        # 使用 self.sampling_params (从rollout config初始化)
        pass
```

**关键发现**:
- ✅ `is_validate=True` 时，使用 `val_kwargs` 的采样参数
- ❌ `is_validate=False` (训练时)，使用默认的 `rollout` 采样参数
- ⚠️ **这导致训练和验证使用不同的采样策略！**

## 二、采样参数差异

### 2.1 训练采样参数来源

**配置位置**: `qwen3_agentloop.sh:L134-147`

```bash
actor_rollout_ref.rollout.max_model_len=32768 \
actor_rollout_ref.rollout.n=8 \               # 训练时每个prompt生成8个response
# 训练时使用的采样参数从rollout config中读取
# (temperature, top_p, top_k 等，没有在脚本中明确指定)
```

**默认行为**:
- 训练时，如果没有设置 `validate=True`，SGLang会使用初始化时的 `self.sampling_params`
- 这些参数通常来自rollout配置的默认值

### 2.2 验证采样参数来源

**配置位置**: 需要在config中设置 `actor_rollout_ref.rollout.val_kwargs`

```yaml
actor_rollout_ref:
  rollout:
    val_kwargs:
      n: 5              # 验证时每个prompt生成5个response
      do_sample: False  # 是否采样
      temperature: 0.7  # 验证专用temperature
      top_p: 0.95      # 验证专用top_p
      top_k: 50        # 验证专用top_k
```

⚠️ **问题**: 你的脚本中没有明确设置 `val_kwargs` 的采样参数！

### 3.3 Reward函数初始化差异

**代码位置**: `verl/trainer/main_ppo.py:282-287`

```python
# 训练reward函数
reward_fn = load_reward_manager(
    config, 
    tokenizer, 
    num_examine=0,  # ⚠️ num_examine=0 (不打印样本)
    **config.reward_model.get("reward_kwargs", {})
)

# 验证reward函数
val_reward_fn = load_reward_manager(
    config, 
    tokenizer, 
    num_examine=1,  # ⚠️ num_examine=1 (打印1个样本用于检查)
    **config.reward_model.get("reward_kwargs", {})
)
```

**关键差异**:
1. **num_examine参数不同**:
   - 训练: `num_examine=0` (不打印样本)
   - 验证: `num_examine=1` (打印1个样本)
   
2. **使用相同的reward_kwargs**: 两者都使用 `config.reward_model.reward_kwargs`

3. **reward manager是否完全相同?**
   - 理论上，除了 `num_examine` 外，两个reward manager应该行为一致
   - 但如果reward manager内部有状态或随机性，可能会有差异

## 五、潜在问题

### 5.1 采样参数未明确配置

⚠️ **问题**: 你的脚本中没有设置 `actor_rollout_ref.rollout.val_kwargs` 的采样参数

**影响**:
- 验证时可能使用不同的采样参数
- 导致训练和验证的行为差异

**建议修复**:
```bash
actor_rollout_ref.rollout.val_kwargs.n=8 \
actor_rollout_ref.rollout.val_kwargs.do_sample=True \
actor_rollout_ref.rollout.val_kwargs.temperature=1.0 \  # 与训练一致
actor_rollout_ref.rollout.val_kwargs.top_p=0.95 \      # 与训练一致
actor_rollout_ref.rollout.val_kwargs.top_k=50 \        # 与训练一致
```

### 5.2 Reward函数可能行为不同

⚠️ **问题**: `reward_fn` 和 `val_reward_fn` 是两个不同的对象

**可能的影响**:
1. **如果reward函数有内部状态**: 两个实例可能有不同的状态
2. **如果reward函数依赖validate标志**: 验证时传递了 `meta_info["validate"]=True`
3. **如果reward函数使用了异步调用**: 训练时可能使用async，验证时不使用

**你的配置**:
```bash
custom_reward_function.path=verl/utils/reward_score/llm_judge_async.py
custom_reward_function.name=compute_score_batch
```

⚠️ **`llm_judge_async.py` 的行为需要检查**:
- 是否依赖 `validate` 标志?
- 是否有内部状态或缓存?
- 训练和验证时的调用方式是否一致?

### 5.3 Agent Loop工具调用可能不同

如果你的reward函数依赖于agent loop的轨迹（包括工具调用结果），那么:

1. **训练时**: 使用默认采样参数 → 可能生成不同的轨迹
2. **验证时**: 使用val_kwargs采样参数 → 可能生成不同的轨迹
3. **结果**: 即使prompt相同，最终的reward可能不同

## 八、总结

### 当前状态
❌ **训练和验证的rollout逻辑不完全一致**:
1. 验证时明确设置了 `validate=True`，训练时没有
2. 验证时可能使用不同的采样参数 (val_kwargs)
3. 两者使用不同的reward函数对象 (虽然初始化参数相同)

### 建议优先级
1. **高优先级**: 添加 `val_kwargs` 的采样参数配置，确保与训练一致
2. **中优先级**: 检查 `llm_judge_async.py` 是否依赖 `validate` 标志
3. **低优先级**: 在训练时也明确设置 `validate=False`

### 最终建议
**如果你想让训练和验证完全一致**，需要:
1. 明确配置 `val_kwargs` 的所有采样参数
2. 确保reward函数不依赖 `validate` 标志
3. 可选: 在训练时也传递 `validate=False`
