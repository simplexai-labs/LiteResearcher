# VERL 配置文件使用指南

## 概述

本文档详细说明了 `agentloop_baseline_TIS.sh` 脚本使用的配置文件结构和加载逻辑。

## 配置文件加载流程

### 1. 脚本执行命令分析

```bash
python3 -m verl.trainer.main_ppo \
    --config-path="$CONFIG_PATH" \
    --config-name='search_multiturn_grpo' \
    # ... 其他参数覆盖
```

**关键参数：**
- `--config-path`: `examples/sglang_multiturn/config` (项目配置目录)
- `--config-name`: `search_multiturn_grpo` (主配置文件名，不含 .yaml 扩展名)

### 2. Hydra 配置加载机制

VERL 使用 [Hydra](https://hydra.cc/) 进行配置管理。配置加载遵循以下层次结构：

```
search_multiturn_grpo.yaml (主配置)
    ↓
ppo_trainer.yaml (基础配置)
    ↓
各个组件的默认配置 (actor, data, rollout, critic, etc.)
    ↓
命令行参数覆盖
```

## 配置文件详解

### 主配置文件

**位置：** `examples/sglang_multiturn/config/search_multiturn_grpo.yaml`

```yaml
hydra:
  searchpath:
    - file://verl/trainer/config  # 指定额外的配置搜索路径

defaults:
  - ppo_trainer  # 继承 verl/trainer/config/ppo_trainer.yaml
  - _self_       # 当前文件的配置优先级最高

data:
  max_prompt_length: 1024
  max_response_length: 1024
  train_batch_size: 256
  return_raw_chat: True
  shuffle: False

actor_rollout_ref:
  hybrid_engine: True
  rollout:
    name: sglang
    multi_turn:
      enable: True
      max_assistant_turns: 2
      format: qwen  # 注意：会被脚本参数覆盖为 hermes
```

### 基础配置文件

**位置：** `verl/trainer/config/ppo_trainer.yaml`

这是所有 PPO 训练的基础配置文件，定义了默认的组件配置：

```yaml
defaults:
  # 组件配置的加载格式：
  # - <组件类型>@<配置路径>: <yaml文件名>

  - actor@actor_rollout_ref.actor: dp_actor
  - data@data: legacy_data
  - ref@actor_rollout_ref.ref: dp_ref
  - rollout@actor_rollout_ref.rollout: rollout
  - model@actor_rollout_ref.model: hf_model
  - critic@critic: dp_critic
  - reward_model@reward_model: dp_reward_model
  - _self_

actor_rollout_ref:
  hybrid_engine: true
  nccl_timeout: 600
  rollout:
    layered_summon: False

algorithm:
  _target_: verl.trainer.config.AlgoConfig
  gamma: 1.0
  lam: 1.0
  adv_estimator: gae
  # ... 其他算法配置

trainer:
  balance_batch: True
  total_epochs: 30
  # ... 其他训练器配置
```

### 组件配置文件

配置文件按功能分类存储在不同子目录中：

#### 1. **Data 配置**
**位置：** `verl/trainer/config/data/legacy_data.yaml`

```yaml
tokenizer: null
train_files: ~/data/rlhf/gsm8k/train.parquet
val_files: ~/data/rlhf/gsm8k/test.parquet
prompt_key: prompt
max_prompt_length: 512
max_response_length: 512
train_batch_size: 1024
return_raw_chat: False
shuffle: True
# ... 其他数据配置
```

#### 2. **Rollout 配置**
**位置：** `verl/trainer/config/rollout/rollout.yaml`

```yaml
_target_: verl.workers.config.RolloutConfig
name: ???  # 必须在上层配置中指定 (sglang/vllm/hf)
mode: sync
temperature: 1.0
top_k: -1
top_p: 1
prompt_length: ${oc.select:data.max_prompt_length,512}
response_length: ${oc.select:data.max_response_length,512}

# Multi-turn 配置
multi_turn:
  _target_: verl.workers.config.MultiTurnConfig
  enable: False
  max_assistant_turns: null
  tool_config_path: null
  max_user_turns: null
  max_parallel_calls: 1
  max_tool_response_length: 256
  tool_response_truncate_side: middle
  format: hermes
  # ... 其他 multi-turn 配置
```

#### 3. **Actor 配置**
**位置：** `verl/trainer/config/actor/dp_actor.yaml`

定义了 actor 模型的训练配置，包括优化器、FSDP 配置等。

#### 4. **Critic 配置**
**位置：** `verl/trainer/config/critic/dp_critic.yaml`

定义了 critic 模型的配置。

#### 5. **Model 配置**
**位置：** `verl/trainer/config/model/hf_model.yaml`

定义了模型加载相关的配置。

## 你的脚本使用的配置

### 最终配置合并顺序

1. **基础配置：** `verl/trainer/config/ppo_trainer.yaml`
2. **组件默认配置：**
   - `data/legacy_data.yaml`
   - `actor/dp_actor.yaml`
   - `rollout/rollout.yaml`
   - `critic/dp_critic.yaml`
   - `reward_model/dp_reward_model.yaml`
   - `model/hf_model.yaml`
   - `ref/dp_ref.yaml`
3. **主配置覆盖：** `examples/sglang_multiturn/config/search_multiturn_grpo.yaml`
4. **命令行参数覆盖：** 脚本中所有 `key=value` 形式的参数

### 关键参数覆盖示例

脚本中的命令行参数会覆盖配置文件中的值：

```bash
# 覆盖 data 配置
data.train_batch_size=512                    # 覆盖 legacy_data.yaml 中的 train_batch_size
data.max_prompt_length=4096                  # 覆盖 legacy_data.yaml 中的 max_prompt_length
data.train_files="$TRAIN_DATA"               # 覆盖训练数据路径

# 覆盖 rollout 配置
actor_rollout_ref.rollout.name=sglang        # 覆盖 rollout.yaml 中的 name
actor_rollout_ref.rollout.multi_turn.format=hermes  # 覆盖 format (从 qwen 改为 hermes)
actor_rollout_ref.rollout.multi_turn.tool_config_path="$TOOL_CONFIG"  # 指定工具配置

# 覆盖 algorithm 配置
algorithm.adv_estimator=grpo                 # 覆盖优势估计器类型
algorithm.rollout_is=${rollout_is}           # 覆盖 IS 配置
```

```yaml

```

## `_generated_ppo_trainer.yaml` 文件的作用

### 文件概述

**位置：** `verl/trainer/config/_generated_ppo_trainer.yaml`

这是一个**自动生成的参考配置文件**，由脚本 `scripts/generate_trainer_config.sh` 生成。

### 主要用途

1. **文档参考**：提供完整的扁平化配置视图，方便开发者查看所有可用的配置选项及其默认值
2. **配置验证**：在 CI/CD 中用于验证配置文件的一致性
3. **调试辅助**：帮助理解 Hydra 合并后的最终配置结构

### 生成过程

```bash
# 生成命令（由 scripts/generate_trainer_config.sh 执行）
python3 scripts/print_cfg.py --cfg job

# 这个命令会：
# 1. 加载 ppo_trainer.yaml 及其所有依赖的配置文件
# 2. 通过 Hydra 合并所有配置
# 3. 将多层嵌套的配置展开为单个扁平化的 YAML 文件
```

### 文件特点

```yaml
# 文件头部说明
# This reference configration yaml is automatically generated via 'scripts/generate_trainer_config.sh'
# in which it invokes 'python3 scripts/print_cfg.py --cfg job ' to flatten the 'verl/trainer/config/ppo_trainer.yaml' config fields into a single file.
# Do not modify this file directly.
# The file is usually only for reference and never used.

# 完整的配置内容（532行）
actor_rollout_ref:
  actor:
    optim:
      _target_: verl.workers.config.FSDPOptimizerConfig
      lr: 1.0e-06
      # ... 所有优化器配置
    fsdp_config:
      # ... 所有 FSDP 配置
    # ... 所有 actor 配置
  # ... 所有其他配置
```

### 重要说明

⚠️ **不要直接修改此文件！**

- 此文件是**只读参考文件**
- 任何修改都会在下次运行生成脚本时被覆盖
- 如需修改配置，应该修改源配置文件：
  - `ppo_trainer.yaml`（基础配置）
  - 各个组件的配置文件（如 `actor/dp_actor.yaml`）
  - 你的主配置文件（如 `search_multiturn_grpo.yaml`）

### 与运行时配置的区别

| 对比项 | `_generated_ppo_trainer.yaml` | 实际运行时配置 |
|--------|-------------------------------|----------------|
| **来源** | 自动生成的静态文件 | Hydra 动态合并 |
| **内容** | 仅基础配置的合并 | 基础配置 + 主配置 + 命令行参数 |
| **用途** | 文档参考、CI验证 | 实际训练使用 |
| **是否可修改** | ❌ 不可修改 | ✅ 通过源文件修改 |

### 如何使用这个文件

**场景1：查找配置项的默认值**
```bash
# 想知道 actor.ppo_mini_batch_size 的默认值
grep -A 2 "ppo_mini_batch_size" verl/trainer/config/_generated_ppo_trainer.yaml
```

**场景2：理解配置结构**
```bash
# 查看完整的 rollout 配置结构
sed -n '/rollout:/,/^[^ ]/p' verl/trainer/config/_generated_ppo_trainer.yaml
```

**场景3：对比配置差异**
```bash
# 对比你的修改和默认配置
diff verl/trainer/config/_generated_ppo_trainer.yaml \
     examples/sglang_multiturn/config/search_multiturn_grpo.yaml
```

### CI/CD 集成

在 CI 流程中，`generate_trainer_config.sh` 会：

1. 重新生成 `_generated_ppo_trainer.yaml`
2. 检查生成的文件是否与 Git 中的版本一致
3. 如果不一致，说明有人修改了源配置但没有重新生成此文件
4. CI 会失败并提示开发者运行生成脚本

```bash
# CI 检查逻辑（来自 generate_trainer_config.sh:44-48）
if ! git diff --exit-code -- "$target_cfg" >/dev/null; then
    echo "✖ $target_cfg is out of date. Please regenerate via 'scripts/generate_trainer_config.sh' and commit the changes."
    exit 1
fi
```

## 常见问题

### Q1: 为什么我修改了某个配置文件但没有生效？

**A:** 检查配置优先级。命令行参数 > 主配置文件 > 基础配置文件 > 组件默认配置。如果命令行有同名参数，会覆盖配置文件中的值。

### Q2: 如何知道某个参数的默认值？

**A:** 按照配置加载顺序查找：
1. 先查看 `ppo_trainer.yaml` 中的 defaults 列表
2. 找到对应组件的配置文件（如 `rollout/rollout.yaml`）
3. 查看该文件中的默认值

### Q3: format=qwen 和 format=hermes 有什么区别？

**A:** 这决定了工具调用的格式解析方式：
- `hermes`: 使用 `<tool_call>...</tool_call>` 格式（由 `HermesToolParser` 解析）
- `qwen`: 使用 Qwen 模型的工具调用格式（可能需要单独的 parser）

你的脚本使用 `format=hermes`，对应的解析器在 `verl/experimental/agent_loop/tool_parser.py:75`
