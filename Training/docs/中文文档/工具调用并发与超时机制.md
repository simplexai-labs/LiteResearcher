# AgentLoop 工具调用并发与超时机制详解

## 概述

本文档详细说明了 VERL AgentLoop 中工具调用的并发控制和超时机制的实现方式。系统采用**五层架构**：**Rollout层**、**AgentLoop层**、**工具层（Tool Layer）**、**执行层（Execution Layer）** 和 **后端服务层（Backend Service Layer）**。

**并发控制的三个层次**：
1. **Rollout层并发**：多个请求并发处理（由SGLang batch能力决定）
2. **AgentLoop层并发**：单个请求内单次轮次的工具并发（`max_parallel_calls`）
3. **执行层并发**：全局工具执行池并发（`num_workers` + `rate_limit`，所有请求共享）

当前示例脚本：`/share/project/wanli/Search_Agent/verl/examples/sglang_multiturn/search_browser/agentloop_search_browse.sh`

## 一、AgentLoop层并发控制

### 1.1 状态机设计

位于 `verl/experimental/agent_loop/tool_agent_loop.py`：


**状态转换流程**：
```
PENDING → GENERATING → PROCESSING_TOOLS → GENERATING → ...
                   ↓                    ↓
              TERMINATED            INTERACTING
```

### 1.2 并发工具调用实现

在 `_handle_processing_tools_state` 方法中：

```python
async def _handle_processing_tools_state(self, agent_data: AgentData) -> AgentState:
    """处理工具调用状态：并发执行多个工具"""

    # 🔑 创建并发任务列表
    tasks = []
    for tool_call in agent_data.tool_calls[: self.max_parallel_calls]:  # 限制并发数
        tasks.append(
            self._call_tool(
                tool_call,
                agent_data.tools_kwargs,
                request_ref=agent_data.request_ref  # 传递链接历史缓冲区
            )
        )

    # 🔑 使用 asyncio.gather 并发执行所有工具
    with simple_timer("tool_calls", agent_data.metrics):
        responses = await asyncio.gather(*tasks)

    # 处理所有工具响应
    for tool_response, tool_reward, _ in responses:
        # 将工具响应添加到消息列表
        add_messages.append({"role": "tool", "content": tool_response.text})
        # ...
```

**并发控制关键点**：
- `max_parallel_calls`：限制**单个请求在单次工具调用轮次中同时执行的工具数量**（默认值：1）
  - ⚠️ **作用域**：单个请求（single request）的单次工具调用轮次（single turn）
  - **示例**：模型某次生成解析出5个工具调用，`max_parallel_calls=3` 则只并发执行前3个
  - **注意**：这是请求级别的并发，不是全局并发
- `asyncio.gather(*tasks)`：Python异步并发执行多个工具调用
- 所有工具调用**并行执行**，然后**统一等待**结果

**与其他并发参数的区别**：
- `max_parallel_calls`：**单个请求内**的工具并发数（请求级别）
- `num_workers`：**工具执行池**的全局并发数（所有请求共享）
- `rate_limit`：**全局限流**的速率控制（所有请求共享）

### 1.3 并发控制的三个层次详解

为了清楚理解整个系统的并发控制，需要区分三个层次：

#### 层次1：Rollout层并发（多个请求并发）

```python
# Rollout过程中，多个请求并发处理
# 例如：GRPO算法，每个prompt采样n=5个响应
batch_size = 512  # 一个batch有512个prompt
n = 5             # 每个prompt采样5个响应
# 总共 512 * 5 = 2560 个请求并发处理

# SGLang会将这些请求分batch并发执行
# 每个请求独立运行一个AgentLoop实例
```

**特点**：
- 由SGLang/vLLM的batch并发能力决定
- 多个请求**完全独立**，互不影响
- 每个请求有自己的AgentLoop状态机

#### 层次2：AgentLoop层并发（单个请求内，单次轮次的工具并发）

```python
# 单个请求的某一次生成中，模型输出了多个工具调用
# 例如：模型生成 [search, browse1, browse2, browse3, browse4]

max_parallel_calls = 3  # 配置值

# 实际执行：只取前3个并发执行
agent_data.tool_calls = [search, browse1, browse2, browse3, browse4]
tasks = []
for tool_call in agent_data.tool_calls[:max_parallel_calls]:  # 只取前3个
    tasks.append(self._call_tool(tool_call, ...))

# 使用asyncio.gather并发执行这3个工具
responses = await asyncio.gather(*tasks)
# 结果：search, browse1, browse2 并发执行
# browse3, browse4 被忽略（超过max_parallel_calls限制）
```

**特点**：
- 作用域：**单个请求的单次工具调用轮次**
- 如果模型输出5个工具调用，`max_parallel_calls=3`，只执行前3个
- 默认值为1（串行执行工具）
- 这是**请求级别**的并发，不同请求之间互不影响

#### 层次3：执行层并发（全局工具执行池并发）

```python
# 所有请求共享的全局工具执行池
class GoogleSearchTool:
    def __init__(self, config):
        self.num_workers = 120      # ExecutionPool的max_concurrency
        self.rate_limit = 120       # TokenBucket的容量

# 全局限制：
# - 同时最多120个search工具在执行（num_workers）
# - 每秒最多发起120个search请求（rate_limit）

# 场景示例：
# 假设有100个请求同时调用search工具
# - 前120个工具调用会立即执行（num_workers限制）
# - 第121个及以后的工具调用会排队等待
# - TokenBucket确保不会瞬间发起超过120个请求
```

**特点**：
- 作用域：**全局（所有请求共享）**
- `num_workers`：Ray Actor的max_concurrency，限制执行池并发数
- `rate_limit`：TokenBucket限流器，限制请求发起速率
- 所有请求**共享**这个限制

---

#### 三个层次的关系示例

假设有以下配置：
- `batch_size=512`，`n=5`（GRPO）
- `max_parallel_calls=3`
- `num_workers=120`，`rate_limit=120`（search工具）

**并发场景分析**：

```
时刻T0：Rollout开始
├─ 512个prompt，每个采样5个响应 = 2560个请求
├─ SGLang batch并发处理这2560个请求
│
时刻T1：某个请求A的模型生成完成
├─ 模型输出5个工具调用：[search1, search2, search3, search4, search5]
├─ max_parallel_calls=3，只取前3个：[search1, search2, search3]
├─ asyncio.gather并发执行这3个工具
│  ├─ search1 → execution_pool.execute.remote()
│  ├─ search2 → execution_pool.execute.remote()
│  └─ search3 → execution_pool.execute.remote()
│
时刻T2：多个请求同时到达执行池
├─ 请求A的3个工具：search1, search2, search3
├─ 请求B的3个工具：browse1, browse2, browse3
├─ 请求C的2个工具：search4, browse4
├─ ... 还有很多其他请求
│
执行层处理（全局限制）：
├─ TokenBucketWorker检查：当前并发数 < rate_limit=120？
│  ├─ 如果是，获取令牌，继续执行
│  └─ 如果否，阻塞等待令牌释放
├─ ExecutionPool检查：当前执行数 < num_workers=120？
│  ├─ 如果是，执行HTTP请求
│  └─ 如果否，排队等待
```

**总结**：
- **层次1（Rollout）**：2560个请求并发处理
- **层次2（AgentLoop）**：每个请求取前3个工具并发执行
- **层次3（执行层）**：全局最多120个工具同时执行

### 1.4 RequestRef链接历史管理

```python
class RequestRef:
    def __init__(self, request_id: str, maxlen: int = 200):
        self.request_id = request_id
        self.link_history_buffer = deque(maxlen=maxlen)  # 循环缓冲区
        self._set = set()  # 快速查找

    def add_links_to_buffer(self, links: list[str]):
        """添加链接到历史缓冲区（由Google Search工具调用）"""
        for link in links or []:
            norm = self._normalize(link)  # 规范化URL
            if norm and norm not in self._set:
                self._set.add(norm)
                self.link_history_buffer.append(norm)

    def is_link_in_buffer(self, link: str) -> bool:
        """验证链接是否在历史缓冲区中（由Browse工具调用）"""
        return self._normalize(link) in self._set
```

**工作原理**：
1. **Google Search工具**：提取搜索结果中的链接，调用 `add_links_to_buffer()` 存储
2. **Browse工具**：调用前检查 `is_link_in_buffer()`，确保链接来自搜索结果
3. **防止幻觉**：避免模型编造不存在的链接

**链接提取逻辑**：
```python
def extract_links_from_search_result(self, search_result: str) -> List[str]:
    """从Google搜索结果中提取链接"""
    links = []
    result_data = json.loads(search_result)
    result_text = result_data.get("result", "")

    # 使用正则表达式匹配 "link: https://...\n" 格式
    link_pattern = r'link:\s*(https?://[^\n]+)\n'
    found_links = re.findall(link_pattern, result_text)

    for link in found_links:
        cleaned_link = link.rstrip('.,;!?')
        if cleaned_link:
            links.append(cleaned_link)

    return links
```

## 三、执行层实现细节

### 3.1 令牌桶限流器（TokenBucketWorker）

位于 `google_search_tool.py` 和 `browse_tool.py`，使用 **信号量（Semaphore）** 实现全局限流：

```python
@ray.remote(concurrency_groups={"acquire": 1, "release": 10})
class TokenBucketWorker:
    """Ray actor for rate limiting using token bucket algorithm."""

    def __init__(self, rate_limit: int):
        self.rate_limit = rate_limit
        self.current_count = 0  # 当前占用的令牌数
        self._semaphore = threading.Semaphore(rate_limit)  # 🔑 限流核心

    @ray.method(concurrency_group="acquire")
    def acquire(self):
        """获取令牌（如果没有可用令牌则阻塞）"""
        self._semaphore.acquire()
        self.current_count += 1

    @ray.method(concurrency_group="release")
    def release(self):
        """释放令牌"""
        self._semaphore.release()
        self.current_count -= 1

    def get_current_count(self):
        """获取当前并发数（用于监控）"""
        return self.current_count
```

**工作原理**：
1. **初始化**：创建容量为 `rate_limit` 的信号量（例如120）
2. **acquire**：获取令牌，如果没有可用令牌则**阻塞等待**
3. **release**：释放令牌，允许其他请求继续
4. **并发组**：
   - `acquire`：串行执行（并发度=1），避免竞争
   - `release`：并行执行（并发度=10），快速释放

### 3.2 执行工作节点（GoogleSearchExecutionWorker）

```python
class GoogleSearchExecutionWorker:
    """Worker for executing search engine operations with optional rate limiting."""

    def __init__(self, enable_global_rate_limit=True, rate_limit=10):
        # 🔑 初始化单例限流器（所有worker共享）
        self.rate_limit_worker = self._init_rate_limit(rate_limit) if enable_global_rate_limit else None

    def _init_rate_limit(self, rate_limit):
        """初始化单例限流器"""
        return TokenBucketWorker.options(
            name="search-engine-rate-limiter",  # 🔑 全局单例名称
            get_if_exists=True                   # 🔑 如果存在则复用
        ).remote(rate_limit)

    def execute(self, fn: Callable[..., T], *fn_args, **fn_kwargs) -> T:
        """执行函数，带限流控制"""
        if self.rate_limit_worker:
            with ExitStack() as stack:
                # 🔑 确保释放令牌（即使异常也会执行）
                stack.callback(self.rate_limit_worker.release.remote)
                # 🔑 获取令牌（阻塞直到获取成功）
                ray.get(self.rate_limit_worker.acquire.remote())
                try:
                    return fn(*fn_args, **fn_kwargs)
                except Exception as e:
                    logger.warning(f"Error when executing: {e}")
                    raise
        else:
            return fn(*fn_args, **fn_kwargs)
```

**执行流程**：
1. **获取令牌**：调用 `rate_limit_worker.acquire()`，如果达到限制则**阻塞**
2. **执行函数**：执行实际的工具调用逻辑
3. **释放令牌**：通过 `ExitStack.callback` 确保令牌释放（**异常安全**）

## 四、HTTP请求超时与重试

### 4.1 Google Search的HTTP超时

位于 `verl/tools/utils/google_search_utils.py`：

```python
DEFAULT_TIMEOUT = 30  # 默认超时30秒
MAX_RETRIES = 3       # 最大重试3次
INITIAL_RETRY_DELAY = 1  # 初始重试延迟1秒

def call_google_search_api(search_service_url: str, query: str,
                          search_type: str = "hybrid", limit: int = 10,
                          timeout: int = DEFAULT_TIMEOUT):
    """调用Google Search API，带重试机制"""

    request_id = str(uuid.uuid4())
    log_prefix = f"[Google Search Request ID: {request_id}] "

    payload = {
        "query": query,
        "search_type": search_type,
        "limit": limit
    }

    last_error = None

    for attempt in range(MAX_RETRIES):
        try:
            logger.info(f"{log_prefix}Attempt {attempt + 1}/{MAX_RETRIES}")

            # 🔑 HTTP请求超时控制
            response = requests.post(
                search_service_url,
                headers={"Content-Type": "application/json"},
                json=payload,
                timeout=timeout,  # 🔑 超时参数
            )

            # 🔑 处理5xx服务器错误，重试
            if response.status_code in [500, 502, 503, 504]:
                if attempt < MAX_RETRIES - 1:
                    delay = INITIAL_RETRY_DELAY * (attempt + 1)  # 递增延迟
                    logger.info(f"{log_prefix}Retrying after {delay} seconds...")
                    time.sleep(delay)
                continue

            response.raise_for_status()
            return response.json(), None

        except requests.exceptions.Timeout as e:
            # 🔑 超时异常，重试
            last_error = f"{log_prefix}Timeout Error: {e}"
            if attempt < MAX_RETRIES - 1:
                delay = INITIAL_RETRY_DELAY * (attempt + 1)
                time.sleep(delay)
            continue

        except requests.exceptions.ConnectionError as e:
            # 🔑 连接错误，重试
            last_error = f"{log_prefix}Connection Error: {e}"
            if attempt < MAX_RETRIES - 1:
                delay = INITIAL_RETRY_DELAY * (attempt + 1)
                time.sleep(delay)
            continue

    # 所有重试失败
    logger.error(f"{log_prefix}API call failed. Last error: {last_error}")
    return None, last_error
```

**超时和重试策略**：
- **请求超时**：30秒（可配置）
- **最大重试**：3次
- **递增延迟**：1秒、2秒、3秒（避免瞬间压垮服务）
- **重试场景**：
  - 5xx 服务器错误
  - 超时（Timeout）
  - 连接错误（ConnectionError）

### 4.2 Browse的HTTP超时

```python
DEFAULT_TIMEOUT = 30  # 默认超时30秒
MAX_RETRIES = 10      # 最大重试10次（Browse需要更多重试）
INITIAL_RETRY_DELAY = 1

def call_browse_api(browse_service_url: str, link: str, what_to_find: str,
                   timeout: int = DEFAULT_TIMEOUT):
    """调用browse API，带重试机制"""

    request_id = str(uuid.uuid4())
    log_prefix = f"[Browse Request ID: {request_id}] "

    payload = {
        "url": link,
        "question": what_to_find
    }

    for attempt in range(MAX_RETRIES):
        try:
            logger.info(f"{log_prefix}Attempt {attempt + 1}/{MAX_RETRIES}")

            # 🔑 HTTP请求超时控制
            response = requests.post(
                browse_service_url,
                headers={"Content-Type": "application/json"},
                json=payload,
                timeout=timeout,
            )

            # 处理5xx服务器错误，重试
            if response.status_code in [500, 502, 503, 504]:
                if attempt < MAX_RETRIES - 1:
                    delay = INITIAL_RETRY_DELAY * (attempt + 1)
                    logger.info(f"{log_prefix}Retrying after {delay} seconds...")
                    time.sleep(delay)
                continue

            response.raise_for_status()
            return response.json(), None

        except (requests.exceptions.Timeout, requests.exceptions.ConnectionError) as e:
            if attempt < MAX_RETRIES - 1:
                delay = INITIAL_RETRY_DELAY * (attempt + 1)
                logger.info(f"{log_prefix}Retrying after {delay} seconds...")
                time.sleep(delay)
            continue

    return None, last_error
```

```python
    # 🔑 负载均衡：节点1(42.9%) + 节点2(42.9%) + 节点3(14.3%)
    rand = random.random()
    if rand < 3/7:
        node_idx = 0
    elif rand < 6/7:
        node_idx = 1
    else:
        node_idx = 2

    vllm_url = VLLM_URLS[node_idx]
    print(f"🔀 使用节点{node_idx + 1}: {vllm_url}")

    # 🔑 Token截断逻辑（避免超长文档）
    MAX_TOKENS = 30000
    if estimate_tokens(text) > MAX_TOKENS - 2000:
        max_chars = (MAX_TOKENS - 2000) * 2
        text = text[:max_chars] + "\n\n[文档过长已截断...]"
        print(f"⚠️ 文档已截断: 保留前{max_chars}字符")

    # 构建prompt
    prompt = f"""Based on the following document content, answer the user's question:

Document Title: {title}
Document Content: {text}

User Question: {question}

Please provide a concise and accurate answer based on the document content:"""

    payload = {
        "model": "/mnt/nas_nfs/home/wanli/Models/Qwen3-4B-Instruct-2507-FP8",
        "messages": [{"role": "user", "content": prompt}],
        "max_tokens": max_tokens,
        "temperature": temperature

    for attempt in range(VLLM_MAX_RETRIES + 1):
        try:
            async with self.session.post(
                f"{vllm_url}/v1/chat/completions",
                json=payload,
                timeout=aiohttp.ClientTimeout(total=VLLM_TIMEOUT)  # 🔑 超时300秒
            ) as response:
                if response.status == 200:
                    data = await response.json()
                    answer = data.get("choices", [{}])[0].get("message", {}).get("content", "")
                    return {"answer": answer, "vllm_time": time.time() - vllm_start}

                # 5xx错误，重试
                elif response.status >= 500:
                    if attempt < VLLM_MAX_RETRIES:
                        print(f"🔄 vLLM HTTP {response.status}, 重试 {attempt + 1}/{VLLM_MAX_RETRIES}")
                        await asyncio.sleep(VLLM_RETRY_DELAY * (attempt + 1))
                        continue

        except (asyncio.TimeoutError, aiohttp.ClientError) as e:
            if attempt < VLLM_MAX_RETRIES:
                print(f"🔄 vLLM {type(e).__name__}, 重试 {attempt + 1}/{VLLM_MAX_RETRIES}")
                await asyncio.sleep(VLLM_RETRY_DELAY * (attempt + 1))
                continue

    return {"answer": "vLLM max retries exceeded", "error": "Max retries exceeded"}
```

## 八、并发与超时参数调优建议

### 8.1 AgentLoop层配置

**max_parallel_calls（单个请求内的工具并发数）**：

```yaml
# 在训练配置中设置
actor_rollout_ref.rollout.multi_turn.max_parallel_calls: 3  # 🔑 推荐值
```

**调优建议**：
- **默认值：1**（串行执行工具，最保守）
- **小值（1-2）**：
  - 适用场景：工具调用成本高、需要严格控制资源
  - 优点：资源消耗低，结果可控
  - 缺点：速度慢，如果模型输出多个工具则执行不完整
- **中等值（3-5）**：⭐ **推荐值**
  - 适用场景：平衡并发和资源消耗
  - 优点：能执行多个工具，提升效率
  - 缺点：需要确保后端服务能承受（考虑Rollout层的并发）
- **大值（>10）**：
  - 适用场景：轻量级工具、高吞吐场景
  - 优点：最大化工具调用效率
  - 缺点：可能超过后端承载能力（需要配合调整num_workers和rate_limit）

**⚠️ 关键考虑**：
- `max_parallel_calls` 是**请求级别**的并发，所有请求的工具调用会在执行层**叠加**
- 计算全局并发压力：`batch_size * n * max_parallel_calls`
  - 例如：`512 * 5 * 3 = 7680` 个工具调用可能同时到达执行层
- 因此需要配合调整 `num_workers` 和 `rate_limit` 来控制全局并发

### 8.2 工具层配置

**num_workers（并发线程数）**：
```yaml
# Google Search工具
num_workers: 120  # 向量搜索速度快，可以高并发

# Browse工具
num_workers: 60   # vLLM推理慢，并发数适中
```

**调优原则**：
- 不超过后端服务的处理能力
- 考虑Ray集群的资源限制
- 监控 `TokenBucketWorker.current_count` 是否一直满载

**rate_limit（限流）**：
```yaml
# 与num_workers保持一致或略小
rate_limit: 120  # Search
rate_limit: 60   # Browse
```

**调优原则**：
- 防止瞬间流量冲击后端服务
- 使用令牌桶算法平滑流量
- 根据后端服务的QPS能力调整

**timeout（HTTP超时）**：
```yaml
# Google Search
timeout: 60   # 向量搜索通常很快

# Browse
timeout: 180  # 需要等待SQL+vLLM，时间较长
```

**调优原则**：
- 根据后端服务的平均响应时间设置
- 留出足够的余量（避免频繁超时）
- 考虑重试次数，超时*重试次数 = 最大等待时间

### 8.4 性能瓶颈诊断

**问题1：工具调用很慢，请求排队**
- **症状**：`TokenBucketWorker.current_count` 一直满载
- **原因**：`num_workers` 或 `rate_limit` 设置过小
- **解决**：增加 `num_workers` 和 `rate_limit`

**问题2：后端服务过载，返回5xx错误**
- **症状**：HTTP请求频繁返回500/503错误
- **原因**：工具层并发数超过后端承载能力
- **解决**：减少 `num_workers` 或增加后端 `workers`

**问题3：请求频繁超时**
- **症状**：HTTP请求频繁超时，重试次数多
- **原因**：`timeout` 设置过小，或后端处理慢
- **解决**：增加 `timeout` 或优化后端性能（如vLLM推理速度）

**问题4：vLLM推理超时**
- **症状**：Browse请求经常超时，vLLM报错
- **原因**：文档过长，token超限
- **解决**：8084.py已实现自动截断（30000 tokens），检查是否生效

## 十、总结

VERL AgentLoop的工具调用系统采用**五层架构**实现并发和超时控制，其中并发控制分为**三个层次**：

### 10.1 五层架构总结

**层次1：Rollout层**（多请求并发）
- SGLang/vLLM batch并发处理多个请求
- 每个请求独立运行AgentLoop
- 并发数由 `batch_size * n` 决定

**层次2：AgentLoop层**（单请求内工具并发）
- 使用 `asyncio.gather` 并发执行多个工具
- ⚠️ **`max_parallel_calls` 限制单个请求单次轮次的工具并发数**
- RequestRef 管理链接历史缓冲区，防止模型幻觉

**层次3：工具层**
- 使用 Ray Remote Actor 实现异步并发
- 链接提取（Search）和链接验证（Browse）机制

**层次4：执行层**（全局并发控制）
- ⚠️ **`num_workers` + `rate_limit` 限制所有请求共享的全局并发**
- TokenBucketWorker 实现全局限流
- HTTP请求超时控制和重试机制

**层次5：后端服务层**
- FastAPI + uvicorn 多进程并发
- aiohttp异步客户端连接池
- 内部超时和重试机制
- 负载均衡（Browse服务3个vLLM节点）

### 10.2 关键优势

- ✅ **高并发**：
  - AgentLoop层：asyncio.gather并发执行工具
  - 工具层：Ray + TokenBucket限流
  - 后端层：uvicorn多进程 + aiohttp连接池

- ✅ **限流保护**：
  - 全局令牌桶限流器
  - 单主机连接数限制
  - 防止瞬间流量冲击

- ✅ **容错性**：
  - HTTP层重试（Browse 10次，Search 3次）
  - 后端层重试（vLLM 3次，Milvus 3次）
  - 递增延迟，避免雪崩

- ✅ **防止幻觉**：
  - RequestRef链接历史缓冲区
  - Browse工具验证链接来源
  - 确保链接来自Search结果

- ✅ **可观测性**：
  - 详细日志（每层都有）
  - 监控端点（/stats, /health）
  - Request ID追踪

- ✅ **灵活配置**：
  - YAML配置文件
  - 易于调整并发、限流、超时参数
  - 支持不同场景的优化

### 10.3 最佳实践

#### 1. 并发参数设置

**AgentLoop层（单个请求内的并发）**：
- `max_parallel_calls`: 3-5（推荐值）
- **考虑因素**：
  - 模型通常输出2-3个工具调用
  - 过大的值（>10）对单个请求意义不大
  - 需要考虑全局并发压力：`batch_size * n * max_parallel_calls`

**工具层（全局执行池并发）**：
- `num_workers`: 根据后端能力
  - Search工具：120（向量搜索快，可以高并发）
  - Browse工具：60（vLLM推理慢，并发数适中）
- `rate_limit`: 与num_workers一致或略小
  - 防止瞬间流量冲击
  - 使用令牌桶算法平滑流量

**全局并发压力计算**：
```python
# 假设配置
batch_size = 512       # 一个batch的prompt数
n = 5                  # GRPO采样数
max_parallel_calls = 3 # 单个请求的工具并发

# 理论最大并发压力
max_global_concurrency = batch_size * n * max_parallel_calls
# = 512 * 5 * 3 = 7680 个工具调用

# 实际并发控制
# 执行层限制：num_workers=120, rate_limit=120
# 实际同时执行：最多120个工具调用
# 其他7560个工具调用会排队等待
```

#### 2. 超时参数设置

- **Search工具**: 60秒（向量搜索快）
- **Browse工具**: 180秒（SQL+vLLM慢）
- **后端vLLM**: 300秒（长文本处理）

#### 3. 重试策略

- **Browse HTTP**: 10次（vLLM不稳定）
- **Search HTTP**: 3次（向量搜索稳定）
- **vLLM推理**: 3次（处理临时错误）

#### 4. 监控指标

- **AgentLoop层**：
  - 平均每个请求调用的工具数量
  - max_parallel_calls的实际使用率
- **执行层**：
  - TokenBucket.current_count（当前并发）
  - HTTP成功率和重试次数
  - 工具调用排队时间
- **后端服务**：
  - QPS和响应时间
  - 错误率和超时率
  - 链接缓冲区命中率（Browse工具）

通过合理配置并发数、限流和超时参数，可以在保证系统稳定性的同时，最大化工具调用的吞吐量，实现高效的多轮Agent交互。
