## 概述

在 PPO/GRPO 训练过程中，verl 会定期在验证集上评估模型性能。验证过程会：

1. 对验证集中的每个问题生成 N 个响应（N 由 `actor_rollout_ref.rollout.val_kwargs.n` 配置）
2. 使用奖励函数对每个响应进行评分
3. 计算各种统计指标（平均值、最佳值、多数投票等）
4. 按数据源分组统计结果

这些指标帮助你了解：
- 模型的平均性能如何
- Best-of-N 采样能达到多高的性能
- 多数投票策略的效果如何
- 不同数据集上的表现差异

## 验证流程

验证过程的关键步骤（位于 `verl/trainer/ppo/ray_trainer.py` 的 `_validate` 函数）：

```python
def _validate(self):
    # 1. 遍历验证数据加载器
    for test_data in self.val_dataloader:
        test_batch = DataProto.from_single_dict(test_data)

        # 2. 重复每个样本 N 次（生成 N 个不同响应）
        test_batch = test_batch.repeat(
            repeat_times=self.config.actor_rollout_ref.rollout.val_kwargs.n,
            interleave=True
        )

        # 3. 生成响应
        test_output_gen_batch = self.actor_rollout_wg.generate_sequences(test_gen_batch)

        # 4. 使用奖励函数评分
        result = self.val_reward_fn(test_batch, return_dict=True)
        reward_tensor = result["reward_tensor"]
        scores = reward_tensor.sum(-1).cpu().tolist()

        # 5. 收集额外信息（如预测答案）
        if "reward_extra_info" in result:
            for key, lst in result["reward_extra_info"].items():
                reward_extra_infos_dict[key].extend(lst)

    # 6. 处理验证指标
    data_src2var2metric2val = process_validation_metrics(
        data_sources, sample_uids, reward_extra_infos_dict
    )

    # 7. 组织指标输出
    metric_dict = {}
    for data_source, var2metric2val in data_src2var2metric2val.items():
        for var_name, metric2val in var2metric2val.items():
            for metric_name, metric_val in metric2val.items():
                # 分类为 val-core 或 val-aux
                # 格式：{metric_sec}/{data_source}/{var_name}/{metric_name}
                pfx = f"{metric_sec}/{data_source}/{var_name}/{metric_name}"
                metric_dict[pfx] = metric_val

    return metric_dict
```

## 指标数据结构

### reward_extra_infos_dict

这是一个字典，收集了每个验证样本的各种信息：

```python
reward_extra_infos_dict = {
    'reward': [0.8, 0.9, 0.7, ...],      # 奖励分数列表
    'pred_ans': ['答案A', '答案B', ...],  # 预测答案列表
    'score': [0.8, 0.9, 0.7, ...],       # 评分列表
    # ... 其他来自 reward_extra_info 的字段
}
```

**关键字段说明：**

- **`reward`**: 模型响应获得的奖励分数（通常是 0 或 1，表示正确/错误）
- **`pred_ans`**: 模型预测的答案（字符串格式）
- **`score`**: 评分结果（通常与 reward 相同，但某些奖励函数可能有细微差异）
- **其他字段**: 取决于具体的奖励函数实现，可能包括中间计算结果、调试信息等

## 指标命名规则

验证指标使用分层命名结构：

```
{metric_section}/{data_source}/{variable_name}/{metric_name}
```

### 1. metric_section（指标分类）

- **`val-core`**: 核心指标
  - 最重要的指标，用于模型选择和性能评估
  - 包括最大采样数（N_max）的 mean、best、maj 指标
  - 通常是主要关注的变量（reward 或 acc）

- **`val-aux`**: 辅助指标
  - 提供额外的统计信息
  - 包括标准差、中间采样数的指标等
  - 用于深入分析和调试

### 2. data_source（数据源）

表示验证数据来自哪个数据集，例如：

- `Browsecomp`: 浏览器操作数据集
- `GAIA`: GAIA 基准测试
- `GPQA`: GPQA 数据集
- `HLE`: HotpotQA Long-form Evidence 数据集
- `WebWalkerQA`: 网页导航问答数据集
- `Xbench`: 跨领域基准测试

### 3. variable_name（变量名）

表示要统计的变量，通常包括：

- **`reward`**: 奖励分数（最常用）
- **`score`**: 评分结果
- **`acc`**: 准确率（如果可用，会作为 core_var）

### 4. metric_name（指标名称）

具体的统计量名称，详见下一节。

## 指标分类

### Bootstrap 采样方法

验证指标使用 **bootstrap 重采样** 方法来估计统计量的稳定性：

```python
def bootstrap_metric(data, subset_size, reduce_fns, n_bootstrap=1000, seed=42):
    """
    执行 bootstrap 重采样来估计指标的统计特性

    参数:
        data: 数据点列表（如 N 个响应的分数）
        subset_size: 每次采样的大小（如 2, 4, 8, ...）
        reduce_fns: 归约函数列表（如 np.max, np.min, majority_vote）
        n_bootstrap: Bootstrap 迭代次数（默认 1000）
        seed: 随机种子

    返回:
        [(mean, std), ...] 每个归约函数的均值和标准差
    """
    # 重复 n_bootstrap 次:
    #   1. 从 data 中有放回地随机抽取 subset_size 个样本
    #   2. 对每个 reduce_fn 计算统计量
    # 返回所有迭代结果的均值和标准差
```

### 指标类型

#### 1. 基础统计量

**`mean@N`** - N 个响应的平均值
```
计算方式: np.mean(var_vals)
含义: 所有 N 个响应的平均奖励
示例: mean@3 = 0.168 表示 3 个响应的平均奖励为 0.168
```

**`std@N`** - N 个响应的标准差
```
计算方式: np.std(var_vals)
含义: N 个响应之间的变异程度
示例: std@3 = 0.073 表示 3 个响应的标准差为 0.073
```

#### 2. Best-of-N 指标

**`best@N/mean`** - Best-of-N 采样的期望性能
```
计算方式: Bootstrap 采样 N 个响应，取最大值，重复 1000 次求均值
含义: 如果从 N 个响应中选择最好的，平均能达到的性能
示例: best@3/mean = 0.222 表示从 3 个响应中选最好的，平均准确率 0.222
```

**`best@N/std`** - Best-of-N 采样的不确定性
```
计算方式: Bootstrap 采样 1000 次，计算最大值的标准差
含义: Best-of-N 策略的稳定性
示例: best@3/std = 0.055 表示 Best-of-3 的结果有一定波动
```

#### 3. Worst-of-N 指标

**`worst@N/mean`** - Worst-of-N 采样的期望性能
```
计算方式: Bootstrap 采样 N 个响应，取最小值，重复 1000 次求均值
含义: 如果从 N 个响应中选择最差的，平均会是多糟糕
示例: worst@3/mean = 0.118 表示从 3 个响应中最差的平均准确率 0.118
```

**`worst@N/std`** - Worst-of-N 采样的不确定性
```
计算方式: Bootstrap 采样 1000 次，计算最小值的标准差
含义: 最差情况的波动范围
示例: worst@3/std = 0.045 表示最差情况也有一定变化
```

#### 4. 多数投票指标（Majority Voting）

**`maj@N/mean`** - 多数投票的期望性能
```
计算方式: Bootstrap 采样 N 个响应，找出最常见的答案，返回该答案对应的奖励
含义: 使用多数投票策略能达到的平均性能
前提: 需要 reward_extra_infos_dict 中包含 'pred' 字段（预测答案）
示例: maj@3/mean = 0.22 表示多数投票平均准确率 0.22
```

**`maj@N/std`** - 多数投票的不确定性
```
计算方式: Bootstrap 采样 1000 次，计算多数投票结果的标准差
含义: 多数投票策略的稳定性
示例: maj@3/std = 0.05 表示多数投票结果较稳定
```

### 采样数量（N）的选择

代码会自动计算多个采样数量：

```python
ns = []
n = 2
while n < n_resps:
    ns.append(n)
    n *= 2
ns.append(n_resps)
# 例如: 如果 n_resps=3, 则 ns = [2, 3]
# 例如: 如果 n_resps=8, 则 ns = [2, 4, 8]
```

5. **`best@3/std: 0.0046`**

## 使用示例

### 1. 在 WandB/TensorBoard 中查看

验证指标会自动记录到配置的日志系统：

```yaml
# 配置文件
trainer:
  logger: ["console", "wandb"]
  val_check_interval: 100  # 每 100 步验证一次
```

**在 WandB 中查看：**
- 导航到 "Charts" 页面
- 搜索 `val-core/` 查看核心指标
- 比较不同 checkpoint 的 `best@N/mean` 来选择最佳模型

### 2. 从日志文件中提取

```python
import re

# 读取日志
with open("output.log") as f:
    log_content = f.read()

# 提取验证指标
pattern = r"'(val-core/[^']+)': ([0-9.]+)"
matches = re.findall(pattern, log_content)

core_metrics = {name: float(value) for name, value in matches}

# 打印各数据集的 Best-of-N 性能
for metric_name, value in sorted(core_metrics.items()):
    if "best@" in metric_name and "/mean" in metric_name:
        print(f"{metric_name}: {value:.4f}")
```

### 3. 比较不同 checkpoint

```python
def compare_checkpoints(ckpt1_metrics, ckpt2_metrics, data_source="GAIA"):
    """比较两个 checkpoint 在特定数据集上的性能"""

    key_metrics = [
        f"val-core/{data_source}/reward/mean@3",
        f"val-core/{data_source}/reward/best@3/mean",
    ]

    print(f"\n{data_source} 数据集性能对比:")
    print(f"{'指标':<50} {'Checkpoint 1':<15} {'Checkpoint 2':<15} {'提升':<10}")
    print("-" * 90)

    for metric in key_metrics:
        v1 = ckpt1_metrics.get(metric, 0)
        v2 = ckpt2_metrics.get(metric, 0)
        improvement = ((v2 - v1) / v1 * 100) if v1 > 0 else 0

        print(f"{metric:<50} {v1:<15.4f} {v2:<15.4f} {improvement:+.2f}%")
```

### 4. 监控训练进度

```python
def analyze_validation_trend(metrics_history):
    """分析验证指标的趋势"""

    # 提取 mean@N 和 best@N
    mean_scores = []
    best_scores = []
    steps = []

    for step, metrics in metrics_history.items():
        steps.append(step)
        mean_scores.append(metrics.get("val-core/GAIA/reward/mean@3", 0))
        best_scores.append(metrics.get("val-core/GAIA/reward/best@3/mean", 0))

    # 计算 Best-of-N 提升百分比
    improvements = [
        (best - mean) / mean * 100 if mean > 0 else 0
        for mean, best in zip(mean_scores, best_scores)
    ]

    print("训练步数 | 平均性能 | Best-of-3 | 提升百分比")
    print("-" * 60)
    for s, m, b, i in zip(steps, mean_scores, best_scores, improvements):
        print(f"{s:<8} | {m:<8.4f} | {b:<9.4f} | {i:+.2f}%")
```

## 核心指标选择规则

在 `_validate` 函数中，核心指标（`val-core`）的选择遵循以下规则：

```python
# 1. 确定核心变量（core_var）
core_var = "acc" if "acc" in var2metric2val else "reward"

# 2. 确定最大采样数（n_max）
n_max = max([int(name.split("@")[-1].split("/")[0]) for name in metric2val.keys()])

# 3. 选择核心指标
for metric_name, metric_val in metric2val.items():
    if (
        (var_name == core_var)  # 必须是核心变量
        and any(metric_name.startswith(pfx) for pfx in ["mean", "maj", "best"])  # 必须是这些前缀
        and (f"@{n_max}" in metric_name)  # 必须是最大采样数
    ):
        metric_sec = "val-core"  # 标记为核心指标
    else:
        metric_sec = "val-aux"   # 否则为辅助指标
```

**示例：**
- 如果有 `mean@2`, `mean@3`, `best@2`, `best@3`, `worst@3`, `std@3`
- 则 `mean@3` 和 `best@3` 会被标记为 `val-core`
- 其他指标会被标记为 `val-aux`

## 最佳实践

### 1. 模型选择策略

**推荐关注的核心指标：**

```yaml
# 按重要性排序
1. val-core/{data_source}/reward/best@N/mean  # Best-of-N 性能（部署时可用）
2. val-core/{data_source}/reward/mean@N       # 平均性能（单次采样）
3. val-core/{data_source}/reward/maj@N/mean   # 多数投票性能（如果可用）
```

**选择策略：**
- **生产环境**: 关注 `best@N/mean`（实际可以使用 Best-of-N 采样）
- **研究对比**: 关注 `mean@N`（单次采样，公平比较）
- **多答案任务**: 关注 `maj@N/mean`（多数投票更鲁棒）

### 2. 验证配置建议

```yaml
# 配置文件示例
actor_rollout_ref:
  rollout:
    val_kwargs:
      n: 3                    # 每个问题生成 3 个响应
      do_sample: false        # 使用贪婪解码（确定性）
      temperature: 0          # 贪婪解码时设置为 0
      top_p: 1.0              # Top-p 采样参数

trainer:
  val_check_interval: 100     # 每 100 步验证一次
  validation_data_dir: ./validation_trajectory  # 保存验证轨迹
```

**建议：**
- **N 的选择**: N=3 或 N=4 通常足够，更大的 N 会增加计算成本
- **do_sample**: 验证时使用 `false`（贪婪解码）保证可重现性
- **temperature**: 验证时设置为 `0` 启用贪婪解码
- **验证频率**: 根据数据集大小调整，小数据集可以更频繁验证

#### 影响贪婪解码的配置参数

以 `examples/sglang_multiturn/search_browser/inference_from_checkpoint_real.sh` 为例，以下参数会影响验证时的采样行为：

```bash
# 训练时采样配置（默认值，通常不需要在推理脚本中设置）
actor_rollout_ref.rollout.temperature=1.0       # 训练时的温度
actor_rollout_ref.rollout.top_p=1.0            # 训练时的 top_p
actor_rollout_ref.rollout.do_sample=True       # 训练时是否采样

# 验证时采样配置（关键！决定推理行为）
actor_rollout_ref.rollout.val_kwargs.n=3       # 每个问题生成 3 个响应
actor_rollout_ref.rollout.val_kwargs.temperature=0   # 贪婪解码（默认值）
actor_rollout_ref.rollout.val_kwargs.top_p=1.0      # Top-p 参数（默认值）
actor_rollout_ref.rollout.val_kwargs.do_sample=False  # 禁用采样（默认值）
```

**关键配置说明：**

1. **`val_kwargs.do_sample`** (最重要)
   - `False`: 使用贪婪解码（确定性，可重现）
   - `True`: 使用采样（随机性，根据 temperature 和 top_p）

2. **`val_kwargs.temperature`**
   - `0`: 强制贪婪解码（即使 do_sample=True 也会退化为贪婪）
   - `> 0`: 控制采样的随机性（值越大越随机）

3. **`val_kwargs.n`**
   - 控制每个问题生成多少个响应
   - 影响 Best-of-N 和多数投票指标的计算

4. **`val_kwargs.top_p`**
   - 核采样参数，控制候选词的累积概率阈值
   - 只在 `do_sample=True` 且 `temperature > 0` 时生效

**默认行为（如果不显式配置）：**

根据 `verl/trainer/config/rollout/rollout.yaml` 的默认值：
- `val_kwargs.do_sample: False`
- `val_kwargs.temperature: 0`
- `val_kwargs.top_p: 1.0`
- `val_kwargs.n: 1`

因此，如果你的推理脚本中没有显式设置这些参数，将自动使用贪婪解码。

#### 贪婪解码在代码中的实现路径

```python
class SGLangHttpServer:
    async def generate(
        self,
        prompt_ids: torch.Tensor,
        sampling_params: dict[str, Any],  # 第 203 行：接收采样参数
        request_id: str,
        image_data: Optional[list[Any]] = None,

        """第 200 行：SGLang 服务器的生成方法"""

        # 第 209-210 行：设置最大生成长度
        max_new_tokens = min(self.config.response_length, self.config.max_model_len - len(prompt_ids) - 1)
        sampling_params["max_new_tokens"] = max_new_tokens

        # 第 211 行：提取 logprobs 参数
        return_logprob = sampling_params.pop("logprobs", False)

        # 第 213-219 行：创建 SGLang 的生成请求 ★★★
        request = GenerateReqInput(
            rid=request_id,
            input_ids=prompt_ids,
            sampling_params=sampling_params,  # ← 传递给 SGLang
            return_logprob=return_logprob,
            image_data=image_data,
        )

        # 第 220 行：调用 SGLang 的 tokenizer_manager 执行生成 ★★★
        output = await self.tokenizer_manager.generate_request(request, None).__anext__()

        # 第 221-229 行：处理输出
        if return_logprob:
            output_token_logprobs = output["meta_info"]["output_token_logprobs"]
            log_probs, token_ids = zip(
                *[(log_prob, token_ids) for log_prob, token_ids, _ in output_token_logprobs], strict=True
            )
        else:
            token_ids = output["output_ids"]
            log_probs = None
        return TokenOutput(token_ids=token_ids, log_probs=log_probs)
```

**作用**：
- 接收 `sampling_params` 字典（包含 `temperature=0`, `top_p=1.0` 等）

- 调用 `tokenizer_manager.generate_request()` 执行实际的推理

1. **解析 sampling_params**：
   ```python

   top_p = sampling_params.get("top_p", 1.0)
   ```

2. **决定采样策略**：
   ```python
   if temperature == 0:
       # 贪婪解码：选择概率最高的 token
       next_token = torch.argmax(logits, dim=-1)
   else:
       # 采样：根据 temperature 和 top_p 进行采样
       logits = logits / temperature
       probs = torch.softmax(logits, dim=-1)
   ```

3. **执行生成**：
   - 如果 `temperature=0`：每次都选择概率最高的 token（贪婪解码）
   - 如果 `temperature>0`：根据概率分布随机采样

## 常见问题

### Q1: 为什么 `reward` 和 `score` 有两个指标？

**A**: 大多数情况下它们相同，但某些奖励函数可能区分：
- `reward`: 用于训练的奖励信号（可能经过缩放或变换）
- `score`: 原始评分（0/1 准确率）

通常关注 `reward` 即可。

### Q2: 为什么没有 `maj@N` 指标？

**A**: 多数投票需要 `reward_extra_infos_dict` 包含 `pred` 字段（预测答案）。如果你的奖励函数没有返回 `pred_ans`，则不会计算多数投票指标。

确保奖励函数返回：
```python
result = {
    "reward_tensor": reward_tensor,
    "reward_extra_info": {
        "pred": pred_answers,  # 必须包含 pred 字段
        "score": scores,
    }
}
```

### Q3: Bootstrap 采样的意义是什么？

**A**: Bootstrap 提供了统计估计的不确定性：
- `best@N/mean`: 告诉你 Best-of-N 的**期望性能**
- `best@N/std`: 告诉你这个估计的**置信度**

如果 `std` 很大，说明性能波动大，需要更多验证样本。

### Q4: 如何调试验证过程？

**A**: 使用 `validation_data_dir` 保存详细轨迹：

```yaml
trainer:
  validation_data_dir: ./validation_trajectory
```

生成的 JSONL 文件包含：
- 输入问题
- 生成的响应
- 预测答案
- 奖励分数
- Ground truth

可以手动检查错误案例。

### Q5: 验证集应该多大？

**A**: 建议：
- **最小**: 每个数据源 100 个样本（统计显著性）
- **推荐**: 每个数据源 500-1000 个样本（稳定估计）
- **大型**: 每个数据源 3000+ 个样本（发布结果）

验证集太小会导致指标波动大，难以判断真实性能。

## 总结

验证指标提供了多维度的模型性能评估：

1. **基础性能**: `mean@N` - 单次采样的平均表现
2. **Best-of-N**: `best@N/mean` - 多次采样选最好的期望性能
3. **多数投票**: `maj@N/mean` - 集成策略的性能
4. **稳定性**: `std@N`, `best@N/std` - 性能的波动范围
5. **最差情况**: `worst@N/mean` - 了解下限
